────────────────────────────────────────────────────────────────
• Monorepo enxuto: raiz com npm scripts (docker, prisma, test:all) e aplicação principal em apps/backend.

3. SUGESTÕES DE MELHORIA (PRIORIZADAS)
────────────────────────────────────────────────────────────────
Alta prioridade
  A) CI/CD: pipeline que execute npm run build, npm test e npm run test:e2e com Postgres + Redis (Compose) e variáveis mínimas de .env.example.
  B) Jest e2e: manter testTimeout elevado OU isolar AppModule em testes mais leves; considerar --maxWorkers=2 em máquinas lentas para reduzir contenção.
  C) Handles abertos: investigar --detectOpenHandles nos testes unitários que disparam Redis/Prisma; garantir app.close() e $disconnect em e2e.
  D) Schema DB vs Prisma: ambientes de desenvolvimento sem migrate geram inconsistências (ex.: colunas faltando). Documentar “db:setup” obrigatório antes dos testes de integração.

Média prioridade
  E) Consolidação PROJETO-CODIGO-COMPLETO.txt: automatizar via script versionado (npm run dump:sources) para não divergir do git; excluir ou truncar package-lock se o ficheiro ficar grande demais.
  F) Segurança: garantir que apps/backend/dist e .env não entram no repositório; revisar commits acidentais de credenciais.
  G) Rate limit /health: já existe exclusão parcial; revisar se /cockpit e /mobile devem ou não contar no limit global conforme política de produção.

Baixa prioridade / evolução
  H) Frontend ou app mobile nativo: monorepo hoje é backend-centric — avaliar apps/frontend ou repositório separado com contratos OpenAPI exportados.
  I) Observabilidade: correlacionar request-id entre cockpit, mobile e auditoria em logs estruturados.
  J) COORDENADOR no RBAC: hoje mapeado conceitualmente a GERENTE — se o negócio exigir, exigiria migration Prisma (fora do escopo das fases “sem migration”).
